# Writing Agent Briefs

An agent brief is the structured GitHub issue comment posted when an issue moves to `ready-for-agent`. The original issue body and discussion remain useful context, but the agent brief is the durable contract an AFK agent should work from.

When posting an agent brief to GitHub, put the AI disclaimer before the heading:

```markdown
> *This was generated by AI during triage.*
```

## Principles

### Durable Over Precise

The issue may sit in `ready-for-agent` for days or weeks. Write the brief so it remains useful after files are renamed, moved, or refactored.

- Do describe interfaces, types, commands, configuration shapes, and behavioral contracts.
- Do name stable public concepts the agent should search for.
- Do not depend on file paths, line numbers, or current implementation layout.
- Do not assume the codebase structure will be unchanged when the agent starts.

### Behavioral, Not Procedural

Describe what the system should do, not step-by-step implementation instructions.

Good:

```markdown
When a user runs `/triage` with no arguments, they should see a summary of issues needing attention.
```

Bad:

```markdown
Open src/triage/handler.ts and add a switch statement near line 150.
```

### Complete Acceptance Criteria

Every brief needs concrete, testable acceptance criteria. Each criterion should be independently verifiable.

Good:

```markdown
- [ ] Running `gh issue list --label needs-triage` returns issues that have been through initial classification.
```

Bad:

```markdown
- [ ] Triage should work correctly.
```

### Explicit Scope Boundaries

State what is out of scope so the agent does not expand into adjacent work.

## Template

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug / enhancement
**Summary:** one-line description of what needs to happen

**Current behavior:**
Describe what happens now. For bugs, this is the broken behavior.
For enhancements, this is the status quo the feature builds on.

**Desired behavior:**
Describe what should happen after the agent's work is complete.
Be specific about edge cases and error conditions.

**Key interfaces:**
- `TypeName` - what needs to change and why
- `functionName()` return type - what it currently returns vs what it should return
- Config shape - any new configuration options needed

**Reproduction notes for bugs:**
State whether reproduction succeeded, failed, or could not be attempted.
Include observed behavior and the suspected code path when known.

**Acceptance criteria:**
- [ ] Specific, testable criterion 1
- [ ] Specific, testable criterion 2
- [ ] Specific, testable criterion 3

**Out of scope:**
- Thing that should not be changed or addressed in this issue
- Adjacent feature that might seem related but is separate
```

## Good Bug Brief

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** bug
**Summary:** Skill description truncation drops mid-word, producing broken output

**Current behavior:**
When a skill description exceeds 1024 characters, it is truncated at exactly 1024 characters regardless of word boundaries. This produces descriptions that end mid-word.

**Desired behavior:**
Truncation should break at the last word boundary before 1024 characters and append `...` to indicate truncation.

**Key interfaces:**
- `SkillMetadata.description` - no type change needed, but the logic that populates it should preserve whole words.
- Any parser that reads skill frontmatter and extracts the description.

**Reproduction notes for bugs:**
Confirmed with a skill description longer than 1024 characters. The generated description ended in the middle of a word.

**Acceptance criteria:**
- [ ] Descriptions under 1024 characters are unchanged.
- [ ] Descriptions over 1024 characters truncate at the last word boundary before 1024 characters.
- [ ] Truncated descriptions end with `...`.
- [ ] The total length including `...` does not exceed 1024 characters.

**Out of scope:**
- Changing the 1024 character limit.
- Supporting multi-line descriptions.
```

## Good Enhancement Brief

```markdown
> *This was generated by AI during triage.*

## Agent Brief

**Category:** enhancement
**Summary:** Preserve rejected feature decisions for future triage

**Current behavior:**
When a feature request is rejected, the issue is closed with a `wontfix` label and a comment. Future similar requests require the maintainer to recall or search for the prior discussion.

**Desired behavior:**
Rejected feature requests should have a durable record that captures the decision, reasoning, and links to prior requests. During future triage, similar requests should surface that prior decision for maintainer confirmation.

**Key interfaces:**
- Rejected-concept record format - must include decision, reason, and prior request links.
- GitHub triage workflow - should check existing rejected-concept records before recommending labels.

**Acceptance criteria:**
- [ ] Closing an enhancement as `wontfix` preserves the decision and rationale.
- [ ] The preserved record includes a link to the closed GitHub issue.
- [ ] Similar future issues surface the previous rejection during triage.
- [ ] The maintainer confirms whether the previous decision still applies.

**Out of scope:**
- Automated semantic matching without maintainer confirmation.
- Reopening previously rejected features.
- Applying the rejected-concept flow to bugs.
```

## Bad Brief

```markdown
## Agent Brief

**Summary:** Fix the triage bug

**What to do:**
The triage thing is broken. Open the handler and fix the function near line 150.
```

This is bad because it has no category, no current or desired behavior, no acceptance criteria, no scope boundaries, and relies on implementation location details that can go stale.
